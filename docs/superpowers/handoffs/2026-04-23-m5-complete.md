## Real promote (non-dry-run) 을 위한 사전 조건

**gh auth 전환이 반드시 필요.** 현재 active 계정이 `kyungjae` (개인)이라 `moloco/msm-portal` 접근 시 `Repository not found` 404가 뜸.

```bash
gh auth switch --user kyungjaeha-moloco
# 혹은 git credential helper 를 moloco 토큰으로 직접 세팅
```

전환 후엔 `git fetch` / `git push` / `gh pr create` 전부 통과할 것. 이건 호스트 환경 설정이라 코드엔 건드리지 않음.

**부가 주의:**
- 호스트 git 에 `user.name/email` 글로벌 설정이 없어도 promote 는 작동. 코드가 `git -c user.name='Playground (<createdBy>)' -c user.email=playground@moloco.inspect am` 주입. 원 저자는 patch 의 `From:` 헤더 유지되므로 PR 은 원저자에게 attribute 됨.
- Promote 성공 후 호스트 HEAD 는 promote 브랜치에 남음. `git checkout main` 수동 필요. (polish 로 남김)
- Promote 실패 시에도 로컬 브랜치는 유지 (사용자 수동 조사용)

## 주의사항 (지뢰)

1. **실제 origin push — 한번이라도 하면 GitHub 에 알림 쏟아질 수 있음.** 팀 리뷰어 ping 을 피하려면 처음엔 `gh pr create --draft` 로 테스트하거나 fork 로 돌릴 것.
2. **promote 트리 divergence** — baseline 은 샌드박스 기동 시점 스냅샷. 그 이후 호스트 main 이 많이 전진했으면 patch 가 merge conflict 내기 시작함. 그땐 skipped[] 에 계속 쌓이고 결과 모달이 표시함 — **자동 복구 안 함** (MVP 정책).
3. **committer identity 가 Playground 식별용** — PR 리뷰어 눈에 "Playground (kyungjae)" 같은 committer 로 보임. 원 저자와 별개. 원한다면 `user.email=<real-user>@moloco.com` 로 override 로직 추가 여지.
4. **`git push --no-verify`** — husky 우회. msm-portal CI가 push 후 돌지, push hook 에서만 도는지 확인 안 함. 필요 시 `--no-verify` 제거 검토.
5. **dry-run 은 `git fetch` 스킵** — 로컬 캐시된 `origin/main` ref 사용. stale 이면 base 가 낡을 수 있음. 사용자에게 안내 필요 시 UI 에 보조 라인 추가.
