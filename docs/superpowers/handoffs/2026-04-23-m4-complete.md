## M5 범위 (v3 plan §9 / §1.2)

### 목표

샌드박스의 playground-<id> 브랜치 누적 커밋을 호스트 msm-portal 에 patch 시퀀스로 적용하고 PR 생성.

### 산출물 (v3 §1.2 그대로)

1. `git format-patch baseline..HEAD` 로 N 개 .patch 추출 (이미 `orchestrator/lib/playground.js` 의 `promotePlayground` 에 스텁 있음 — M5에서 완성)
2. 호스트 msm-portal (`SOURCE_WORKSPACE_ROOT/msm-portal`) 에서 `git fetch` → 새 브랜치 `playground-<id>-<shortDate>` 생성 (from latest main)
3. `git am <patches>` — 실패 시 해당 patch 스킵 + 로그 (MVP 정책)
4. `git push origin <new branch>`
5. `gh pr create --base main --head <new branch>`
6. PR URL 을 Playground 상태에 저장 (`serializePlayground` 에 `promotedPrUrl` 필드 추가)

### UI 통합

- `PlaygroundDetail` 상단 툴바 또는 에디터 상단에 "🚀 Promote" 버튼
- 진행 상태 모달 (patches 추출 → apply → push → PR)
- 실패한 patch 목록을 사용자에게 노출 (어떤 파일에서 막혔는지)

### 예상 소요

v3 §9 기준 **2.5~3일**. 대부분 `git am` 실패 복구 UX + msm-portal 호스트 clone 의 최신 main 상태 추적.

## 주의사항 (지뢰)

1. **실제 msm-portal 에 push 됨** — M5 구현 중 테스트할 때 origin 에 진짜 브랜치가 생김. `--dry-run` 플래그 반드시 사용하거나 별도 테스트 remote (예: 포크) 로.
2. **`git am` 실패 처리** — husky pre-commit / prettier 가 걸려 있으므로 `--no-verify` 꼭 넣기. msm-portal 의 `.husky` 는 호스트 clone에도 적용됨.
3. **baselineCommitSha ↔ 호스트 main 간 divergence** — 샌드박스 baseline은 copy-in 시점의 스냅샷. 그 이후 호스트 main 이 전진했으면 patch 가 merge conflict 낼 수 있음. `git am` 실패 패턴에 포함.
4. **9cb08297 은 구 이미지** — picker runtime 없음. M5 테스트는 `v3-playground-m3` 태그 playground 로.
5. **playground-app 배포 대상 아님** — dev 서버 (`pnpm dev`) 기준. 프로덕션 빌드는 현재 해본 적 없음.
