### [TWITTER]
"버튼 클릭하면 로딩되고 성공하면 완료 표시요."

이 한 문장 안에 상태가 5개다.

idle → loading → success → error → disabled

state-machines.json으로 정리하니까 AI가 빠트리는 상태가 없어졌다.

### [LINKEDIN]
디자이너가 "로딩 중에는 버튼 비활성화해주세요"라고 말할 때, AI는 몇 개의 상태를 고려할까요?

저의 경험상, 명시하지 않으면 2-3개입니다. idle, loading, done. 하지만 실제 UI에는 훨씬 많은 상태가 존재합니다. error, disabled, partial-success, retry...

이 문제를 해결하기 위해 `state-machines.json`을 만들었습니다. 각 컴포넌트가 가질 수 있는 모든 상태와 전이 조건을 JSON으로 구조화한 파일입니다.

AI에게 이 파일을 읽히고 컴포넌트 구현을 요청하면, 먼저 상태 목록을 확인하고 누락 없이 처리합니다. "error 상태는 어떤 메시지를 보여줄까요?"라고 먼저 묻기 시작했습니다.

인터랙션 설계를 텍스트가 아닌 구조로 만들었을 때 생기는 차이입니다.

여러분은 컴포넌트 상태를 어떻게 문서화하고 계신가요?
