# AI와 로드맵 짜기, 쇼케이스 준비하기

## TL;DR

- M1-M16 마일스톤을 하루 만에 AI와 함께 설계했다.
- 쇼케이스 데모는 실패하지 않는 태스크로만 구성한다. 텍스트 변경 > 스타일 > 컴포넌트 추가 순서.
- 팀 소개 문서에 데모 흐름을 넣은 것이 가장 효과적인 온보딩이었다.

---

## 로드맵을 짜는 방식이 바뀌었다

전통적인 로드맵 작업은 이렇게 간다. 팀 미팅에서 아이디어를 모은다. 각 아이디어를 분석한다. 의존성을 파악한다. 타임라인을 잡는다. 최소 며칠이 걸린다.

Inspect 로드맵은 하루에 만들었다. Claude와 대화하면서.

시작은 이 질문이었다. "내가 만들고 싶은 제품이 있어. Chrome Extension으로 UI 요소를 선택하고, AI가 Docker 샌드박스에서 코드를 수정하고, 라이브 프리뷰를 보여주는 것. 이걸 1인 프로젝트로 만든다면 어떤 순서로 접근해야 해?"

Claude는 질문을 되돌려줬다. "Phase 1에서 사용자가 할 수 있어야 하는 가장 핵심적인 것 하나가 뭔가요?" 이 질문이 전체 설계의 방향을 잡았다.

핵심은 "사용자가 UI 요소를 선택하고, 자연어로 변경을 요청하고, 결과를 보는 것"이었다. 이 하나의 흐름이 완성되면 Phase 1이다. 나머지는 Phase 2, 3으로 밀었다.

## M1-M16: 마일스톤을 정의하는 방법

16개 마일스톤은 "사용자가 할 수 있게 되는 것" 기준으로 정의했다. 기능 기준이 아니다.

예를 들어 M3은 이렇게 정의했다. "PM이 Chrome Extension을 설치하고, 웹 페이지에서 UI 요소를 클릭했을 때 그 요소의 정보가 화면에 표시된다." 구체적이고 검증 가능하다. M3이 완료됐는지 판단하는 데 모호함이 없다.

이런 식으로 M1부터 M16까지 정의하고 나면, 각 마일스톤 사이의 의존성이 자연스럽게 보인다. M5는 M3이 완료되어야 시작할 수 있고, M7은 M4와 M6이 병렬로 진행 중일 때 시작할 수 있다는 식으로.

Claude는 이 의존성 분석을 잘했다. "M5와 M6은 공통 Orchestrator API를 공유하니까, 그 API 설계를 먼저 확정하지 않으면 두 마일스톤 모두 나중에 수정이 생길 수 있어요"라는 식의 지적이 실제로 중요한 순서 변경으로 이어졌다.

70일 계획이 나온 것도 이 과정에서였다. 각 마일스톤에 현실적인 시간을 배분하면 70일. 결과적으로 18일이 됐지만, 70일이라는 기준선이 있었기 때문에 진행 속도를 측정할 수 있었다.

## 쇼케이스 데모를 설계하는 원칙

Phase 1 완료 후 팀에 소개하는 쇼케이스를 준비했다. 데모를 설계하면서 하나의 원칙을 세웠다. 실패하지 않는 태스크만 보여준다.

이건 제품을 숨기는 게 아니다. 제품의 현재 상태를 정확하게 보여주는 것이다. 지금 이 제품이 잘 하는 것을 잘 보여주는 것이 더 정직한 커뮤니케이션이다.

태스크를 신뢰도 순서로 나열하면 이렇다.

**텍스트 변경** — 신뢰도 최상. "이 버튼 레이블을 'Get Started'에서 'Start Now'로 바꿔줘." AI가 정확히 그 텍스트만 바꾸고 다른 건 건드리지 않는다. 결과가 항상 예측 가능하다.

**스타일 변경** — 신뢰도 높음. "이 카드의 배경색을 조금 더 밝게 해줘." 디자인 시스템 내의 변경이면 시스템 컨텍스트를 활용해서 적절한 값을 선택한다.

**컴포넌트 추가** — 신뢰도 중간. 새 컴포넌트를 올바른 위치에 추가하는 건 더 많은 맥락이 필요하다. 쇼케이스에서는 이걸 마지막에 배치하되, "이건 아직 개선 중인 부분"이라고 명시했다.

데모 흐름은 이 순서를 따랐다. 확실한 것부터 시작해서 점점 복잡도를 높인다. 관객의 신뢰를 먼저 쌓고, 미완성 부분을 보여주는 순서.

## 팀 소개 문서에 데모 흐름을 넣은 이유

쇼케이스 전에 팀에 보내는 소개 문서를 만들었다. 보통 이런 문서는 "이 제품이 무엇인지"를 설명한다. 나는 거기에 데모 흐름을 그대로 넣었다.

이유는 단순하다. 사람들이 제품을 이해하는 가장 빠른 방법은 그것을 쓰는 장면을 보는 것이다. 아무리 잘 쓴 설명도 동작하는 제품보다 설득력이 약하다.

문서 구조는 이렇게 만들었다.

1. 문제: PM이 겪는 지금 상황을 한 문단으로
2. 솔루션: "이걸 어떻게 해결하는지" 흐름을 단계별로
3. 데모 태스크: 쇼케이스에서 보여줄 태스크 목록과 예상 결과
4. 현재 상태: 잘 되는 것과 아직 개선 중인 것

4번이 중요하다. "무엇이 아직 안 되는지"를 명시하는 것이 오히려 신뢰를 높인다. 감추면 쇼케이스에서 그게 나왔을 때 신뢰가 깨진다. 미리 말하면 기대를 조율할 수 있다.

이 문서가 실제로 쇼케이스보다 더 많은 대화를 만들었다. 사람들이 문서를 읽고 질문을 가져왔다. 쇼케이스는 그 질문에 답하는 자리가 됐다.

## AI와 로드맵을 짠다는 것

AI와 로드맵을 짜는 게 어색하게 들릴 수 있다. 로드맵은 팀의 맥락, 비즈니스 우선순위, 이해관계자의 기대가 반영되는 것이기 때문이다. AI가 그 맥락을 어떻게 알겠냐고.

맞다. AI는 조직 맥락을 모른다. 그래서 내가 그 맥락을 제공하는 역할을 했다. "이 프로젝트의 목적은 PM의 자율성을 높이는 것이고, 엔지니어 팀의 부담을 줄이는 게 부수적 목표야. 그리고 이건 1인 프로젝트야, 엔지니어 리소스가 없어." 이 맥락을 주면 AI의 제안이 훨씬 현실적이 됐다.

AI가 잘 하는 것은 구조화였다. 내가 머릿속에 흩어져 있는 아이디어를 말하면, AI가 그것들 사이의 논리적 관계를 찾아냈다. 의존성, 순서, 리스크. 이걸 빠르게 외재화하는 데 AI가 효과적이었다.

결국 로드맵의 내용은 내 판단이었다. AI는 그 판단을 더 빠르게 할 수 있도록 구조를 제공했다.

## 재현 가능한 패턴

AI와 로드맵을 짜고 쇼케이스를 준비하면서 굳어진 패턴이다.

첫째, 마일스톤을 기능이 아닌 사용자 행동으로 정의한다. "이게 완료되면 사용자가 무엇을 할 수 있는가."

둘째, 의존성을 먼저 파악한다. 순서가 잘못되면 나중에 큰 수정이 생긴다.

셋째, 쇼케이스 태스크는 신뢰도 높은 것부터 시작한다. 확실한 것으로 신뢰를 쌓고, 개발 중인 것을 마지막에 보여준다.

넷째, 소개 문서에 "안 되는 것"을 명시한다. 기대를 조율하는 것이 신뢰를 만든다.

다섯째, AI에게 조직 맥락을 먼저 제공한다. 맥락 없는 AI의 제안은 일반적이고, 맥락이 있으면 구체적이 된다.
