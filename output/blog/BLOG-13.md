## TL;DR

문서 사이트의 UX 디테일은 작지만 전문성을 드러낸다. 디자인 토큰을 JSON으로 관리하고, Shiki로 5개 언어만 선택적으로 로드하고, 다크 모드를 CSS custom properties로 구현하고, ⌘K 검색을 달았다. 각각의 결정에는 이유가 있다.

---

## 배경

프로덕트 디자이너가 직접 문서 사이트를 개발한다는 건 특이한 일이다. 보통은 개발자가 한다. 하지만 나는 UX와 코드 품질을 동시에 신경 쓸 수 있는 위치에 있었고, 이 사이트가 "그냥 동작하는 문서"가 아니라 "쓰고 싶은 문서"가 되길 바랐다.

쓰고 싶은 문서는 작은 디테일에서 만들어진다.

---

## 디자인 토큰: JSON이 진실의 단일 소스

디자인 토큰을 JSON으로 관리하는 건 이제 업계 표준이 됐다. 하지만 "왜"를 이해하지 못하면 형식만 따르게 된다.

토큰을 JSON으로 관리하면 세 가지가 달라진다.

**1. 디자이너와 개발자가 같은 값을 쓴다.** Figma Variables와 코드의 CSS 변수가 같은 소스에서 나온다. "Figma에서는 `#0F62FE`인데 코드에서는 `#1560BD`야" 같은 불일치가 없어진다.

**2. 테마 전환이 선언적이 된다.** `color-bg-primary`가 라이트 모드에선 `#ffffff`, 다크 모드에선 `#161616`이라는 게 JSON에 명시된다. CSS에서 하드코딩으로 분기하지 않는다.

**3. 문서화가 자동화된다.** JSON을 파싱해서 토큰 문서 페이지를 자동 생성할 수 있다. 토큰이 추가되면 문서도 자동으로 업데이트된다.

색상, 타이포그래피, 스페이싱 토큰을 JSON 스키마로 정의하고, 빌드 타임에 CSS custom properties로 변환했다. 이 파이프라인을 한 번 만들어두면 이후 토큰 추가는 JSON에 한 줄 추가하는 것으로 끝난다.

## Shiki: 5개 언어만 선택적으로 로드하는 이유

코드 하이라이팅 라이브러리로 Shiki를 골랐다. Prism.js나 highlight.js도 있었지만, Shiki가 TextMate 문법을 쓰기 때문에 VS Code와 동일한 하이라이팅 결과를 낸다. 개발자들은 VS Code 색상에 이미 익숙하다.

문제는 번들 크기였다. Shiki는 모든 언어 문법과 테마를 포함하면 수 MB가 된다. 문서 사이트 로딩 속도를 그냥 죽이는 수준이다.

해결책은 선택적 로드다. 우리 문서 사이트에서 실제로 사용하는 언어는 5개였다.

- TypeScript (대부분의 컴포넌트 코드)
- TSX (JSX 예제)
- CSS (스타일 예제)
- JSON (토큰 예제)
- Bash (설치 명령어)

Shiki는 사용할 언어와 테마를 명시적으로 지정해서 초기화할 수 있다. 이렇게 하면 번들에 5개 언어의 문법만 포함되고, 나머지는 제외된다.

번들 최적화는 UX 최적화다. 로딩이 빠른 사이트는 사용자가 "이 사이트 빠르다"고 의식하지 않는다. 그냥 기다림 없이 쓴다.

## 다크 모드 전환: CSS custom properties의 힘

다크 모드 구현 방법은 여러 가지가 있다.

**잘못된 방법**: JavaScript로 각 엘리먼트의 색상을 직접 변경한다. 유지보수 지옥.

**괜찮은 방법**: Tailwind의 `dark:` 프리픽스로 클래스 기반 전환. 하지만 클래스가 폭발적으로 늘어난다.

**좋은 방법**: CSS custom properties + `data-theme` 속성.

구현은 이렇다. `html` 엘리먼트에 `data-theme="light"` 또는 `data-theme="dark"` 속성을 붙인다. CSS에서 각 테마에서의 시맨틱 토큰 값을 정의한다.

```css
[data-theme="light"] {
  --color-bg-primary: #ffffff;
  --color-text-primary: #161616;
}

[data-theme="dark"] {
  --color-bg-primary: #161616;
  --color-text-primary: #f4f4f4;
}
```

컴포넌트는 항상 시맨틱 토큰만 참조한다. `background: var(--color-bg-primary)`. 테마가 바뀌면 CSS custom properties 값이 바뀌고, 모든 컴포넌트가 자동으로 새 색상을 쓴다.

전환 애니메이션도 CSS 변수와 `transition: background-color 200ms ease`로 처리한다. 뚝 바뀌는 게 아니라 부드럽게 전환되는 느낌. 이 200ms 차이가 "완성된 느낌"을 만든다.

## 작은 UX 디테일이 전문성을 보여준다

문서 사이트에서 신경 쓴 작은 것들이 있다.

**URL로 탭 상태 저장**: "Style 탭을 보다가 링크 공유하면 상대방도 Style 탭이 열린다." 이걸 구현하려면 탭 상태를 URL 쿼리 파라미터에 넣어야 한다. 귀찮지만 공유가 많은 문서 사이트에서는 필수다.

**코드 블록 복사 버튼**: 호버 시에만 나타나는 복사 버튼. 항상 보이면 시각적 노이즈가 된다. 코드 블록에 마우스를 올렸을 때만 나타나는 게 맞다.

**Props 테이블의 타입 강조**: `string | number | boolean` 같은 타입에서 `|`를 구분자로 각 타입에 개별 스타일을 적용했다. 읽기가 훨씬 쉬워진다.

**빈 상태 메시지**: 검색 결과가 없을 때 "검색 결과가 없습니다"보다 "'{query}'에 대한 결과가 없습니다. Components 페이지에서 전체 목록을 확인하세요."가 훨씬 유용하다.

이런 디테일 하나하나는 사용자가 의식하지 않는다. 하지만 이게 없으면 사이트가 "미완성"처럼 느껴진다. 전문성은 이 디테일들의 총합이다.

---

## 인사이트

문서 사이트 UX는 사용자가 "뭔가 불편하다"고 말하기 전에 불편함을 제거하는 일이다. 검색이 없으면 찾기 힘들고, 다크 모드가 없으면 눈이 피로하고, 코드 복사가 불편하면 직접 드래그해서 복사한다.

각각의 불편함은 작다. 하지만 쌓이면 "이 문서 사이트 쓰기 싫다"가 된다.

---

## 패턴

- **토큰 = 단일 진실 소스**: JSON → CSS variables 파이프라인
- **번들 최적화 = UX 최적화**: Shiki에서 실제로 쓰는 5개 언어만 로드
- **CSS variables + data-theme**: 가장 깔끔한 다크 모드 구현 패턴
- **작은 디테일의 합**: URL 탭 상태, 호버 복사 버튼, 빈 상태 메시지
