## TL;DR

Inspect는 6단계 사용자 여정을 가진다. Chrome Extension에서 엘리먼트를 선택하고 변경을 설명하면, AI 에이전트가 계획을 세우고 실행하고 PR까지 올린다. 이 과정에서 사용자의 불안을 줄이는 게 설계의 핵심이었다.

---

## 배경

AI 코딩 에이전트를 만들면서 가장 어려운 건 기술이 아니었다. 사용자가 "AI가 지금 뭘 하고 있는지" 모를 때 느끼는 불안을 어떻게 다루냐가 진짜 문제였다.

일반 UI에서는 버튼을 누르면 즉시 결과가 나온다. 1초 이상 걸리면 로딩 스피너를 보여주면 된다. 하지만 AI 에이전트는 다르다. 코드 분석부터 PR 생성까지 수십 초에서 수 분이 걸린다. 이 시간 동안 사용자가 "지금 실행되고 있는 건가? 멈춘 건가? 뭔가 잘못된 건가?"를 생각하게 하면 안 된다.

이 문제를 풀기 위해 전체 사용자 여정을 6단계로 정의하고, 각 단계에서 사용자가 느낄 감정을 먼저 상상한 다음 UI를 설계했다.

---

## 6단계 사용자 여정 맵

**1단계: Select (선택)**
Chrome Extension 사이드 패널에서 사용자가 변경하고 싶은 화면 요소를 클릭한다. Extension은 해당 엘리먼트의 CSS 셀렉터, DOM 위치, 현재 스타일을 수집한다.

이 단계에서 사용자의 감정: 기대감. "이제 뭔가 바뀌겠구나."
설계 포인트: 선택된 요소를 명확하게 하이라이트한다. 파란색 테두리. 요소 이름과 위치도 사이드 패널에 표시한다. "맞아, 이거야"를 즉시 확인할 수 있어야 한다.

**2단계: Describe (설명)**
사용자가 변경 사항을 자연어로 입력한다. "버튼 색상을 파란색으로 변경해줘", "헤더 폰트 크기를 16px로 줄여줘".

이 단계에서 사용자의 감정: 기대 반 불안 반. "AI가 내 말을 이해할까?"
설계 포인트: 예시 문장을 플레이스홀더로 보여준다. 구체적으로 쓸수록 더 정확하다는 힌트를 준다. 글자 수 제한은 두지 않는다. 사용자가 원하는 만큼 설명할 수 있어야 한다.

**3단계: Plan (계획 확인)**
AI가 변경 계획을 세우고 사용자에게 보여준다. "어떤 파일을 어떻게 바꿀 예정입니다."

이 단계에서 사용자의 감정: 검토와 판단. "AI가 올바른 방향으로 가고 있나?"
설계 포인트: Inspect Agent 캐릭터가 분석 결과를 보여준다. 이해한 것, 접근 방법, 단계별 실행 계획, 예상 리스크를 구조화해서 표시한다. 사용자는 "승인"하거나 "수정 요청"할 수 있다.

**4단계: Working (실행 중)**
AI 에이전트가 실제로 코드를 수정하고 있는 단계. 가장 긴 대기 시간이 여기에 있다.

이 단계에서 사용자의 감정: 불안. "멈춘 건 아니겠지? 얼마나 더 기다려야 하지?"
설계 포인트: Progress stepper가 핵심이다. Setup → Working → Verify → Done의 4단계 스테퍼. 현재 어느 단계인지, 각 단계에서 구체적으로 뭘 하고 있는지를 실시간으로 보여준다. "파일 분석 중...", "CSS 변경 적용 중...", "빌드 확인 중..." 같은 세부 상태.

**5단계: Verify (검증)**
변경된 코드가 빌드되고 동작하는지 확인하는 단계.

이 단계에서 사용자의 감정: 기다림 속 안도. "거의 다 왔다."
설계 포인트: 검증 단계는 "끝이 보인다"는 신호다. 스테퍼에서 4단계 중 3단계가 채워지면 사용자는 안도한다. 검증 실패 시에는 3회 재시도 루프를 자동으로 실행하고, 사용자에게 "재시도 중입니다 (1/3)"를 보여준다.

**6단계: Done (완료)**
PR이 생성되고 링크가 전달된다. 사용자는 PR을 검토하고 승인하거나 추가 수정을 요청한다.

이 단계에서 사용자의 감정: 성취감. "내가 코드를 몰라도 이걸 해냈다."
설계 포인트: 완료 화면은 축하해야 한다. PR 링크 + 변경된 파일 목록 + 라이브 프리뷰 링크. 그리고 "또 다른 변경을 요청하시겠어요?" CTA.

## Progress Stepper: "기다리는 시간"의 UX

AI 에이전트 UX에서 가장 중요한 단일 요소는 Progress Stepper다.

사람은 기다림 자체보다 "얼마나 더 기다려야 하는지 모르는 것"을 더 힘들어한다. 공항에서 수하물이 나오기를 기다릴 때, 컨베이어 벨트가 움직이는 걸 보면 불안이 줄어드는 것과 같은 원리다.

스테퍼는 세 가지 정보를 동시에 전달한다.
1. 전체 단계가 몇 개인가 (끝이 어디인가)
2. 지금 어느 단계인가 (지금 어디 있는가)
3. 각 단계에서 구체적으로 뭘 하는가 (멈추지 않고 진행 중인가)

이 세 가지가 모두 충족될 때 사용자는 기다릴 수 있다. 하나라도 빠지면 불안이 커진다.

---

## Approve vs Request Changes: 3회 재시도 루프

AI 에이전트가 틀릴 수 있다는 걸 사용자는 안다. 중요한 건 틀렸을 때의 경험을 어떻게 설계하느냐다.

Plan 단계에서 사용자가 "Request Changes"를 누르면, AI 에이전트는 피드백을 반영해서 새 계획을 제시한다. Working 단계에서 검증이 실패하면 자동으로 최대 3회 재시도한다. 3회 모두 실패하면 사용자에게 알리고 수동 개입을 요청한다.

이 루프 설계에서 중요한 건 사용자가 "나는 계속 통제권을 갖고 있다"고 느끼게 하는 것이다. AI가 알아서 재시도하는 건 편리하지만, 사용자가 완전히 배제되면 불안해진다. 그래서 재시도 횟수와 상태를 항상 보여준다.

---

## 각 단계에서 불안을 줄이는 법

정리하면, 각 단계의 불안 포인트와 해결책은 이렇다.

| 단계 | 불안 | 해결책 |
|------|------|--------|
| Select | "내가 선택한 게 맞나?" | 즉각적인 하이라이트 + 이름 표시 |
| Describe | "AI가 이해할까?" | 예시 문장 + 구체성 힌트 |
| Plan | "AI가 맞는 방향으로 가나?" | 구조화된 계획 + 명시적 승인 단계 |
| Working | "멈춘 건 아닌가?" | Progress stepper + 실시간 상태 |
| Verify | "성공할까?" | 재시도 횟수 표시 + 단계 진행 |
| Done | "제대로 됐나?" | 라이브 프리뷰 + 변경 파일 목록 |

---

## 인사이트

AI 에이전트의 UX는 "AI가 무엇을 하는가"보다 "사용자가 무엇을 느끼는가"를 중심으로 설계해야 한다. 기술이 아무리 뛰어나도 사용자가 불안하면 그 기술은 쓰이지 않는다.

각 단계에서 사용자의 감정을 먼저 정의하고, 그 감정을 다루는 UI를 설계하는 것. 이게 AI 에이전트 UX 설계의 핵심이다.

---

## 패턴

- **감정 우선 설계**: 기능 명세 전에 각 단계의 감정 지도를 그린다
- **Progress Stepper = 불안 억제제**: 전체 단계 + 현재 위치 + 세부 상태 세 가지 모두
- **통제권의 착각**: 자동화하되 상태는 항상 보여준다
- **실패 경험 설계**: 재시도 횟수와 상태를 명시적으로 표시한다
