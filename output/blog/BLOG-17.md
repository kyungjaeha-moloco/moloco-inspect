# AI 분석 결과를 'Inspect Agent' 캐릭터로 보여준 이유

## TL;DR

- LLM의 raw JSON 출력을 그대로 보여주면 아무도 읽지 않는다. Understanding → Approach → Steps → Risks → Verification 구조로 카드 UI를 만들었다.
- LLM이 실패하거나 JSON이 잘릴 때를 대비해 intent별 스마트 템플릿 fallback을 짰다. AI가 없어도 계획이 나온다.
- AI를 "도구"가 아닌 "동료"로 포지셔닝하는 순간 사용자의 관계가 달라진다.

---

## 배경: JSON이 화면에 뜨던 날

Inspect의 핵심 흐름에는 AI 분석 단계가 있다. 사용자가 크롬 익스텐션으로 UI 요소를 선택하고 "이 버튼 색상을 바꿔줘"라고 요청하면, 실제 코딩 에이전트가 달려들기 전에 먼저 Claude Sonnet이 그 요청을 분석한다.

어떤 파일을 건드려야 하는지, 어떤 위험이 있는지, 어떻게 검증할지. 이 분석은 PM이 "맞아, 이게 내가 원하는 거야"를 확인하고 진행을 승인하는 체크포인트다.

처음에는 그냥 분석 결과를 텍스트로 보여줬다. API 응답의 `analysis` 필드 내용을 그대로 `<pre>` 태그에 넣었다. 어차피 PM이 다 읽을 거라고 생각했다.

그런데 실제로 사용해보니 아무도 읽지 않았다. 텍스트 덩어리가 화면에 뜨면 눈이 빠르게 훑고 "뭔가 분석했나 보다"라고 느끼는 것이 전부였다. 세부 내용은 스킵됐다.

이건 사용성 문제가 아니었다. 정보 구조의 문제였다.

---

## 문제: LLM 출력은 사람의 읽기 패턴을 고려하지 않는다

Claude에게 실행 계획을 생성하라고 프롬프트를 보내면 JSON이 돌아온다.

```json
{
  "understanding": "사용자가 버튼 색상 변경을 요청했습니다...",
  "analysis": "src/apps/tving/components 경로에서...",
  "steps": ["1. 컴포넌트 파일 찾기", "2. 색상 토큰 확인", ...],
  "risks": "색상 토큰 변경 시 다른 컴포넌트에 영향 가능",
  "verification": "시각적으로 색상 변경 확인"
}
```

이 JSON을 그대로 보여주는 건 레스토랑 주방에서 요리 레시피를 손님에게 건네는 것과 같다. 정보는 있지만 전달이 안 된다.

PM이 실제로 알고 싶은 것은 순서가 있다. 먼저 "AI가 내 요청을 제대로 이해했는가"를 확인하고 싶다. 그다음 "어떻게 할 계획인가"를 보고 싶다. 그리고 마지막으로 "뭘 조심해야 하는가"를 체크하고 싶다. 이 인지 순서가 JSON 구조와 맞지 않았다.

---

## 해결: Understanding → Approach → Steps → Risks → Verification

카드 UI를 만들었다. Inspect Agent가 말을 건네는 형식이다.

```
Inspect Agent
─────────────────────────────────────
이해한 내용
  "버튼 배경색을 brand/primary 토큰에서 danger/primary로 변경..."

접근 방법
  "src/apps/tving/components/Button 파일에서 color prop을..."

실행 단계
  ① 컴포넌트 파일 탐색
  ② 색상 토큰 확인
  ③ Props 수정
  ④ TypeScript 검증
  ⑤ 시각적 확인

위험 요소
  ⚠ 색상 토큰 공유 시 다른 컴포넌트에 영향 가능

검증 방법
  프리뷰에서 버튼 색상이 올바르게 변경됐는지 확인
─────────────────────────────────────
```

각 섹션은 PM의 인지 순서와 일치한다. "이해한 내용"을 읽고 "맞아"라고 느끼면 계속 읽는다. "위험 요소"는 눈에 띄는 위치에 있어서 놓치지 않는다. "실행 단계"는 번호가 있어서 진행 상황을 추적할 수 있다.

같은 데이터다. 구조가 달라졌을 뿐이다.

---

## 또 다른 문제: LLM은 가끔 실패한다

프로덕션에서 AI 분석이 없으면 어떻게 되는가. LLM API 호출이 실패하거나, 응답이 너무 늦거나, 반환된 JSON이 잘려있을 때.

처음에는 그냥 에러를 보여줬다. "분석 실패. 다시 시도해주세요." 사용자는 멈춘다.

이건 제품의 흐름을 끊는 최악의 UX다. 분석이 없어도 코딩 에이전트는 실행될 수 있다. 분석은 체크포인트이지 필수 전제조건이 아니다.

그래서 두 가지를 만들었다.

**첫째, 잘린 JSON 자동 복구.**

LLM이 `max_tokens` 한도에 걸려서 JSON이 중간에 잘리는 경우가 있다. 특히 `steps` 배열이 길 때 발생했다. 이걸 처리하는 로직을 짰다.

```javascript
// 열린 따옴표 수 세기
const quotes = (fixed.match(/"/g) || []).length;
if (quotes % 2 !== 0) fixed += '"';

// 닫히지 않은 배열/객체 닫기
for (let i = 0; i < openBrackets - closeBrackets; i++) fixed += ']';
for (let i = 0; i < opens - closes; i++) fixed += '}';
```

중괄호, 대괄호, 따옴표의 개수를 세서 불균형이 있으면 자동으로 닫아준다. 완벽하진 않지만, 잘린 JSON의 상당 부분을 살려낸다.

**둘째, intent별 스마트 템플릿 fallback.**

LLM 분석이 완전히 실패했을 때를 위해 intent 유형별 템플릿을 만들었다. `layout_adjustment`, `state_handling`, `copy_update`, `component_swap` — 네 가지 intent 유형마다 맥락에 맞는 계획이 자동 생성된다.

```javascript
const intentMap = {
  layout_adjustment: {
    understanding: `${elementInfo} 요소의 레이아웃/배치 변경을 요청했습니다.`,
    steps: [
      `${pageLabel} 페이지 관련 컴포넌트 파일 탐색`,
      `${elementInfo}의 현재 Flex/Grid 구조 분석`,
      `레이아웃 속성 수정`,
      `주변 요소와의 정렬/간격 검증`,
      `TypeScript 검증 및 시각적 확인`,
    ],
    // ...
  },
  copy_update: { /* i18n 파일 경로까지 포함한 맞춤 계획 */ }
};
```

사용자는 LLM이 살아있는지 죽어있는지 모른다. 항상 Inspect Agent가 계획을 내놓는다. 그게 LLM이 만든 건지, 템플릿이 만든 건지는 중요하지 않다.

---

## 왜 "Inspect Agent"라는 이름인가

이 부분이 가장 의식적인 UX 결정이었다.

AI 분석 결과를 보여주는 방법은 여러 가지다. "AI 분석 결과", "분석 완료", "실행 계획". 이런 헤더 아래 내용을 보여줄 수 있다.

하지만 이름을 붙이는 것과 안 붙이는 것은 다르다.

이름 없는 AI 출력은 도구의 출력이다. 복사기가 뱉어낸 문서처럼, 확인하고 치워야 하는 것이다. 이름 있는 AI 출력은 협력자의 제안이다. 읽고, 동의하거나 반박하고, 대화할 수 있는 것이다.

Inspect Agent가 "이 요청을 이렇게 이해했습니다"라고 말할 때, PM은 "맞아" 또는 "아니야, 내가 원한 건 그게 아니야"라고 반응할 수 있다. 이 반응이 제품을 더 쓸 만하게 만든다. AI의 실수를 "AI가 틀렸다"가 아니라 "Inspect Agent가 이 부분을 잘못 이해했다"로 표현할 수 있게 된다.

AI를 도구가 아닌 동료로 포지셔닝하는 것. 그게 Inspect Agent의 존재 이유다.

---

## 핵심 인사이트

AI 출력의 품질은 모델의 성능만으로 결정되지 않는다. 그 출력을 어떻게 구조화하고, 어떻게 보여주고, 어떤 이름으로 부르느냐도 품질의 일부다.

LLM이 완벽한 JSON을 내놓아도 UI가 그것을 텍스트 덩어리로 보여주면 사용자는 읽지 않는다. LLM이 실패해도 fallback이 충분히 유용하면 사용자는 계속 신뢰한다.

AI 제품의 UX는 AI와 사용자 사이의 번역 레이어다.

---

## 재현 가능한 패턴

AI 분석 결과를 UI에 보여줄 때 이 순서로 설계한다.

첫째, 사용자의 인지 순서를 먼저 정의한다. 사용자가 가장 먼저 알고 싶은 게 뭔지, 그다음이 뭔지, 마지막이 뭔지. 그 순서에 따라 섹션을 배치한다.

둘째, LLM 실패 시나리오를 미리 설계한다. fallback 없이 배포하면 AI가 잠깐 죽을 때마다 제품 전체가 죽는다. intent 유형별 템플릿은 최소한의 안전망이다.

셋째, AI에게 이름을 붙인다. "AI 분석"보다 구체적인 이름이 사용자와 AI 사이의 관계를 바꾼다. 도구가 동료가 되는 순간이다.
