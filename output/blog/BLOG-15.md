## TL;DR

코드 diff는 개발자도 읽기 어렵다. PM과 디자이너를 위한 코드 리뷰 UI를 설계할 때, "diff를 이해할 필요가 없게 만든다"는 원칙을 세웠다. 2컬럼 레이아웃, sticky action bar, 타임라인 색상 코딩, 스크린샷과 AI 요약이 그 해결책이었다.

---

## 배경

Inspect는 AI 에이전트가 코드를 수정하고 PR을 올려준다. 그런데 PR을 검토하는 사람이 꼭 개발자일 필요는 없다. 오히려 "이 UI 변경이 디자인대로 됐나?"를 판단하는 건 PM이나 디자이너가 더 잘할 수 있다.

문제는 코드 diff다. GitHub PR 페이지에서 diff를 보면 개발자도 맥락 없이는 이해하기 힘들다. PM이나 디자이너에게 diff를 보여주면 "이게 뭔가요?"가 된다.

그래서 Inspect Hub의 Request Detail 페이지는 처음부터 "코드를 모르는 사람이 변경 사항을 검토할 수 있게"를 목표로 설계했다.

---

## 2컬럼 레이아웃: 왼쪽과 오른쪽의 역할 분리

Request Detail 페이지는 두 개의 컬럼으로 나뉜다.

**왼쪽 컬럼 (기술적 정보)**
- Diff viewer: 어떤 코드가 어떻게 바뀌었는지
- Changed files: 몇 개의 파일이 수정됐는지, 각 파일 이름

왼쪽은 개발자를 위한 영역이다. 코드를 읽을 수 있는 사람은 여기서 정확히 무엇이 바뀌었는지 확인한다.

**오른쪽 컬럼 (맥락과 판단)**
- Before/After 스크린샷: 변경 전후의 실제 화면
- AI 분석 요약: Inspect Agent가 설명하는 "이 변경이 무엇인가"
- 라이브 프리뷰 링크: 실제로 동작하는 결과물
- Sticky Action Bar: Approve 또는 Request Changes 버튼

오른쪽은 비개발자를 위한 영역이다. 스크린샷으로 "Before에는 버튼이 빨간색이었고, After에는 파란색이다"를 즉시 알 수 있다. AI 요약으로 "왜 이런 변경이 일어났는가"를 이해한다. 라이브 프리뷰로 "실제로 내가 요청한 대로 됐는가"를 직접 확인한다.

이 구조는 의도적이다. 왼쪽은 "어떻게 바뀌었나"의 증거고, 오른쪽은 "제대로 바뀌었나"의 판단 근거다. 두 가지는 다르다.

---

## Sticky Action Bar: 스크롤해도 항상 접근 가능

Request Detail 페이지는 콘텐츠가 많다. Diff가 길어지면 페이지 전체 높이가 상당해진다. 사용자가 내용을 다 읽고 맨 아래 내려갔더니 Approve 버튼이 없다면? 다시 스크롤 올라가서 버튼을 찾아야 한다.

이건 설계 실수다.

오른쪽 컬럼의 하단을 sticky로 처리했다. 사용자가 어디를 스크롤하든, Action Bar는 항상 화면 우측 하단에 고정되어 있다.

Sticky의 위치도 중요하게 설계했다. 왼쪽 전체 너비에 걸친 floating bar보다 오른쪽 컬럼 하단에 고정되는 게 맞다고 판단했다. 이유는 맥락 연결이다. Action Bar는 오른쪽의 스크린샷, AI 요약, 라이브 프리뷰와 같은 맥락에 있다. "이것들을 다 봤으니 이제 결정하세요"라는 흐름이 자연스럽다.

Approve와 Request Changes 버튼의 시각적 위계도 신경 썼다. Approve가 Primary(파란 배경), Request Changes가 Secondary(테두리만). 사용자의 기본 행동 경로는 "확인하고 승인"이기 때문이다.

## 타임라인 이벤트별 컬러 코딩

Request Detail 페이지 하단에는 타임라인이 있다. AI 에이전트가 이 요청을 처리하면서 일어난 모든 이벤트의 기록이다.

타임라인 이벤트는 성격이 다르다.
- 시스템 이벤트: 에이전트가 시작됨, 파일 분석 완료, 빌드 성공
- 사용자 이벤트: 요청 생성, 승인, 수정 요청
- 오류 이벤트: 빌드 실패, 재시도 발생
- 성공 이벤트: PR 생성 완료

이 이벤트들을 같은 색상으로 표시하면 타임라인이 단순한 로그가 된다. 중요한 것과 참고용이 구분이 안 된다.

컬러 코딩을 도입했다.
- **파란색**: 시스템 진행 이벤트 (중립적 진행 상황)
- **초록색**: 성공 이벤트 (PR 생성, 빌드 성공)
- **빨간색**: 오류 이벤트 (빌드 실패, 재시도)
- **보라색**: 사용자 행동 이벤트 (승인, 수정 요청)

사용자는 타임라인을 처음부터 끝까지 읽지 않는다. 색상으로 스캔한다. 빨간색이 있으면 "어, 오류가 있었나?" 하고 그 항목을 읽는다. 초록색이 마지막에 있으면 "성공했구나"를 빠르게 파악한다.

타임라인의 목적은 "무슨 일이 있었는가"의 기록이 아니라 "지금 이 요청의 상태는 괜찮은가"의 빠른 판단이다. 컬러 코딩이 그 목적을 위해 존재한다.

## "PM이 diff를 이해할 필요는 없다"

이 페이지를 설계하면서 가장 중요했던 원칙을 한 문장으로 정리하면 이렇다.

> "PM이 diff를 이해할 필요는 없다. 스크린샷과 AI 요약이면 충분하다."

코드 리뷰 UI를 만들 때의 흔한 실수는 "코드를 잘 보여주는 것"을 목표로 삼는 것이다. 하지만 비개발자를 포함한 모든 이해관계자가 리뷰에 참여해야 한다면, 목표는 "변경의 의도와 결과를 이해하게 하는 것"이어야 한다.

diff는 변경의 메커니즘이다. 스크린샷은 변경의 결과다. AI 요약은 변경의 의도다. 비개발자에게 필요한 건 메커니즘이 아니라 결과와 의도다.

물론 개발자도 이 페이지를 쓴다. 그래서 diff viewer를 왼쪽에 유지한다. 개발자는 diff를 보고, 비개발자는 스크린샷과 AI 요약을 본다. 두 그룹이 같은 페이지에서 각자 필요한 정보를 찾는다.

## Diff Viewer: Carbon Light 테마의 선택 이유

Diff viewer에는 Carbon Light 테마를 적용했다. 사이트 전체가 다크 모드를 지원하는데, diff viewer만 Light 테마로 고정한 이유가 있다.

diff는 빨간색(삭제)과 초록색(추가)으로 변경 사항을 표시한다. 다크 배경에서 이 색상들은 채도가 높아 보여서 강렬하게 느껴진다. 특히 많은 변경이 있을 때 시각적으로 압도된다.

Light 배경에서의 파스텔 빨간색과 파스텔 초록색은 같은 정보를 훨씬 읽기 편하게 전달한다. 코드를 읽는 행위 자체가 인지적으로 부담이 큰데, 배경까지 어두우면 그 부담이 더 커진다.

또한 코드 에디터의 기본 색상 연상이 있다. VS Code의 기본 테마는 Dark이지만, 코드 diff를 보여줄 때는 GitHub처럼 Light 배경이 업계 관행이다. 사용자는 이 패턴에 이미 익숙하다.

## 인사이트

"비개발자를 위한 UI"를 만든다는 건 개발자 기능을 빼는 게 아니다. 비개발자도 같은 정보를 다른 언어로 이해할 수 있게 번역해주는 것이다.

diff는 코드 언어다. 스크린샷과 AI 요약은 시각적 언어와 자연어다. 같은 변경 사항을 세 가지 언어로 표현하면, 어떤 배경의 사용자도 자신에게 맞는 언어로 이해할 수 있다.

---

## 패턴

- **2컬럼 역할 분리**: 왼쪽(기술적 증거) + 오른쪽(판단 근거)
- **Sticky Action Bar**: 판단 버튼은 항상 접근 가능해야 한다
- **컬러 코딩 타임라인**: 읽는 것이 아니라 스캔하게 만든다
- **세 가지 언어**: diff(코드) + 스크린샷(시각) + AI 요약(자연어)
- **배경색은 가독성**: Diff viewer에 Light 테마를 고정한 이유
