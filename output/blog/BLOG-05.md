# 디자이너의 영역이 서버, Docker, CI/CD까지 확장되는 이유

## TL;DR

- Zscaler 프록시 SSL 문제, Docker secret, opencode.json 타임아웃. 디자이너가 이런 걸 겪었다.
- 깊은 기술 지식이 아니라 "어디서 문제가 생겼는지 판단하는 능력"이 필요하다.
- AI 시대에 디자이너의 역할은 "문제를 정의하고 AI에게 해결을 지시"하는 것으로 확장된다.

---

## 디자이너가 Docker를 만나는 날

Docker 컨테이너가 시작되지 않았다. 에러 메시지는 이랬다.

```
Error response from daemon: failed to create shim task: OCI runtime create failed
```

나는 이 메시지가 무슨 뜻인지 몰랐다. Docker를 배운 적이 없었다. OCI runtime이 뭔지도 몰랐다.

하지만 이 에러가 어디서 나왔는지는 판단할 수 있었다. Docker 데몬이 컨테이너를 실행하려고 하는데 뭔가 실패했다는 것. 그리고 이 실패가 네트워크 문제인지, 설정 문제인지, 권한 문제인지를 좁혀가는 것. 이게 내가 할 수 있는 일이었다.

Claude에게 에러 메시지를 가져다줬다. "이 에러가 뭔지, 어떻게 해결하는지 알려줘." Claude가 진단했다. 원인은 macOS에서 Docker Desktop의 virtualization 설정 문제였다. 해결 방법을 알려줬고, 5분 만에 컨테이너가 올라왔다.

이게 내 패턴이 됐다. 문제를 발견한다. 어느 레이어에서 생긴 문제인지 판단한다. AI에게 가져간다. AI가 해결한다.

## Zscaler가 SSL을 막을 때

회사 네트워크는 Zscaler를 쓴다. 기업 프록시다. Docker 샌드박스 안에서 외부 패키지를 설치하려면 npm install을 실행해야 하는데, Zscaler가 SSL 인증서를 중간에서 교체하기 때문에 인증서 검증이 실패했다.

에러는 이랬다.

```
npm ERR! code UNABLE_TO_GET_ISSUER_CERT_LOCALLY
```

처음에는 이게 무슨 문제인지 감이 없었다. npm이 왜 실패하는지. 네트워크 문제인지, Docker 설정 문제인지, npm 문제인지.

단계적으로 좁혔다. Docker 컨테이너 밖에서 npm install이 되는가? 된다. 그럼 Docker 안의 문제다. Docker 컨테이너 안에서 curl로 외부 주소에 접근되는가? 안 된다. 그럼 네트워크 또는 SSL 문제다.

여기서 Claude에게 물었다. "Docker 컨테이너 안에서 curl이 SSL 에러를 내고, 회사 네트워크는 Zscaler를 쓰는데, 이 두 가지가 관련이 있을 수 있어?" Claude가 연결고리를 설명했다. Zscaler가 SSL 트래픽을 가로채고 자체 인증서로 교체하는데, Docker 컨테이너는 그 인증서를 모른다. 해결책은 Zscaler CA 인증서를 Docker 이미지에 주입하는 것.

코드는 Claude가 썼다. 나는 Zscaler CA 인증서가 어디 있는지 찾아서 전달했고, Docker 빌드 과정에 인증서를 주입하는 방법을 Claude가 구현했다.

이 해결에서 내가 기여한 것은 "어느 레이어에서 문제가 생겼는가"를 판단한 것이었다. 코드를 쓴 건 AI였다.

## opencode.json 타임아웃과 에이전트 설정

Docker 샌드박스 안에서 AI 에이전트(opencode)가 코드를 수정하는데, 가끔 타임아웃이 났다. 에이전트가 응답을 안 하고 멈춰버리는 상황.

처음에는 네트워크 문제라고 생각했다. Zscaler 이슈가 또 생긴 건가. 그런데 네트워크는 정상이었다. 에이전트가 특정 조건에서만 멈추는 패턴이 있었다. 복잡한 코드 수정 요청일 때 더 자주 멈췄다.

로그를 보니 opencode가 응답을 기다리는 시간 제한이 기본값으로 설정되어 있었다. 복잡한 요청은 그 시간 안에 끝나지 않아서 에이전트가 죽는 것이었다.

opencode.json 설정 파일에서 타임아웃 값을 조정했다. 이 설정 파일의 존재를 몰랐지만, Claude에게 "opencode 에이전트가 타임아웃으로 죽는 문제가 있는데, 설정으로 조정할 수 있을까?"라고 물었고 파일 경로와 수정 방법을 알려줬다.

설정 값 하나 바꾸는 일이었지만, 그게 어디에 있는지, 어떤 파라미터인지 모르면 찾는 데 시간이 많이 걸렸을 것이다.

## Docker Secret과 보안 설계

Docker 샌드박스 안에서 AI 에이전트가 API 키를 써야 했다. API 키를 어떻게 전달하는가가 문제였다.

처음에는 환경 변수로 넘기려고 했다. 간단하고 직관적이다. Claude가 멈췄다. "환경 변수로 API 키를 넘기면 docker inspect 명령으로 노출될 수 있어요. Docker secret을 쓰는 게 더 안전해요."

나는 Docker secret을 몰랐다. 하지만 "API 키가 노출될 수 있다"는 게 뭘 의미하는지는 알았다. 그게 심각한 문제라는 것도 알았다.

Claude가 설명했다. Docker secret은 컨테이너 런타임에 메모리로만 전달되고 디스크에 기록되지 않는다. 노출 경로가 훨씬 좁다.

구현은 Claude가 했다. 나는 이 결정이 왜 맞는지 이해했고, 그 이해 위에서 다음 설계 결정을 내릴 수 있었다. "그럼 다른 credential도 전부 Docker secret으로 관리하자."

기술의 세부사항은 몰라도 된다. 그 기술이 해결하는 문제와 트레이드오프를 이해하면 된다.

## 디자이너가 이런 걸 알아야 하는가

솔직하게 답하면: 알 필요는 없다. 하지만 판단할 수 있어야 한다.

내가 Zscaler CA 인증서를 Docker에 주입하는 방법을 외울 필요는 없다. 이 문제가 다시 생기면 Claude에게 다시 물으면 된다. 하지만 "이 문제가 네트워크 레이어에서 생겼다"고 판단하는 것, "Zscaler 프록시가 개입하고 있다"고 연결하는 것, 이건 내가 해야 한다.

왜냐하면 AI는 내가 어디를 보고 있는지 모르기 때문이다. 내가 "뭔가 안 돼"라고 말하면 AI는 무한한 가능성에서 원인을 찾아야 한다. 내가 "Docker 컨테이너 안에서 SSL 에러가 나고 있어, 회사 네트워크는 Zscaler를 써"라고 말하면 AI는 정확한 범위에서 해결책을 찾는다.

문제를 정의하는 능력. 어디서 무엇이 잘못됐는지를 좁히는 능력. 이게 AI 시대에 디자이너에게도 필요한 스킬이다.

## 영역이 확장되는 이유

왜 디자이너의 영역이 서버, Docker, CI/CD까지 확장되는가.

이건 선택이 아니다. 제품 전체를 혼자 만들면 자연스럽게 모든 레이어를 다루게 된다. 그 각 레이어에서 문제가 생기고, 그 문제를 해결해야 제품이 앞으로 나간다.

과거에는 이 문제들이 "엔지니어가 할 일"이었다. 디자이너는 그걸 엔지니어에게 넘겼다. 그런데 지금은 AI가 엔지니어 역할의 상당 부분을 할 수 있다. 디자이너가 문제를 정의해서 AI에게 주면, AI가 해결한다.

이 구조에서 디자이너는 더 이상 "Figma 안에서만" 일할 수 없다. 제품 전체를 이해하고, 각 레이어에서 생기는 문제를 식별하고, AI를 통해 해결하는 사람이 된다.

이건 디자이너가 엔지니어가 되는 것이 아니다. 디자이너의 시야가 넓어지는 것이다.

## 깊은 지식 vs 판단 능력

18일 동안 배운 가장 중요한 구분이다.

깊은 기술 지식: Docker의 내부 동작 원리, SSL 핸드셰이크 과정, CI/CD 파이프라인 설계. 이걸 디자이너가 배울 필요는 없다.

판단 능력: "이 에러가 어느 레이어에서 생겼는가", "이 결정이 보안에 영향을 주는가", "이 설정이 성능에 영향을 주는가". 이건 디자이너도 가질 수 있고, 가져야 한다.

그 판단 능력을 키우는 방법은 문제를 많이 겪는 것이다. Zscaler 문제를 겪고 나서, 다음번에 회사 네트워크에서 Docker를 쓸 때 "아, SSL 인증서 문제일 수 있다"고 연결할 수 있다. 코드를 못 써도 이 연결은 할 수 있다.

AI가 코드를 쓴다. 나는 문제를 정의하고 결정을 내린다. 이 분업이 작동하려면 내가 판단력을 가져야 한다. 그 판단력을 키우는 것, 그게 AI 시대 디자이너의 숙제다.

## 재현 가능한 패턴

기술 문제를 만났을 때 내가 쓰는 접근법을 정리하면 이렇다.

첫째, 레이어를 좁힌다. 네트워크 문제인가, 애플리케이션 문제인가, 설정 문제인가. 단계적으로 제거해간다.

둘째, AI에게 맥락을 준다. 에러 메시지만 주지 않는다. "어떤 환경에서, 어떤 작업을 하다가, 어떤 에러가 났는지"를 모두 준다.

셋째, 해결 방법의 이유를 이해한다. AI가 해결책을 줄 때 "왜 이 방법인가"를 물어본다. 이해하면 다음 비슷한 상황에서 판단할 수 있다.

넷째, 트레이드오프를 기록한다. 이번 결정이 무엇을 얻고 무엇을 포기했는지. 이 기록이 나중에 다른 결정의 맥락이 된다.

디자이너의 영역이 넓어진다는 건 무서운 일이 아니다. AI가 옆에 있으면 모르는 것도 해결할 수 있다. 필요한 건 용기와 판단력이다.
