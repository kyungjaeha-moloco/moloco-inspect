## TL;DR

IBM Carbon 디자인 시스템을 레퍼런스로 삼아, ⌘K 글로벌 검색과 다크 모드를 갖춘 문서 사이트를 하루 만에 만들었다. 핵심은 "잘 만든 레퍼런스를 고르는 것"과 "CSS 변수 60개를 미리 정의해두는 것"이었다.

---

## 배경

Inspect의 컴포넌트가 20개를 넘어가면서 문제가 생겼다. 새 팀원이 컴포넌트를 쓰려면 소스 코드를 직접 열어봐야 했다. Storybook을 붙이는 방법도 있었지만, 우리 프로젝트에는 과했다. 필요한 건 딱 세 가지였다.

1. 컴포넌트 목록과 Props 문서
2. 실제 동작하는 코드 예제
3. 다크 모드 지원

그래서 문서 사이트를 직접 만들기로 했다. 하루 안에.

---

## 왜 Carbon인가

레퍼런스를 고를 때 기준은 하나였다. "이 사이트를 보면 내가 만들고 싶은 게 뭔지 알 수 있나?" Carbon Design System은 그 기준을 완벽하게 충족했다.

Carbon이 특별한 이유는 구조의 명확함이다. 좌측 사이드바에 컴포넌트 목록, 우측에 문서 본문, 그리고 본문 안에 Usage → Style → Code → Accessibility 탭. 이 구조는 사용자가 "지금 내가 어디 있는지"를 항상 알게 해준다.

shadcn/ui도 훌륭하지만 카드 기반 레이아웃이라 컴포넌트가 많아질수록 탐색이 어렵다. Ant Design은 너무 무겁다. MUI는 Material 디자인 언어가 너무 강해서 우리 스타일에 맞지 않았다.

결국 Carbon의 레이아웃 구조만 가져오고, 색상과 타이포그래피는 우리 토큰으로 교체했다. 이게 핵심이었다. 레이아웃은 빌리고, 정체성은 우리 것으로.

---

## ⌘K 글로벌 검색

문서 사이트에서 검색은 선택이 아니라 필수다. 컴포넌트가 30개가 넘어가면 사이드바 스크롤로는 원하는 걸 찾기가 힘들다.

`cmdk` 라이브러리를 골랐다. 이유는 단순하다. Linear, Vercel, Raycast가 다 이걸 쓴다. 검증된 UX 패턴을 따를 때는 잘 만든 오픈소스 라이브러리를 그냥 쓰는 게 맞다.

구현에서 신경 쓴 부분은 검색 데이터 구조였다. 컴포넌트 이름, 설명, 태그를 모두 인덱싱해서 "버튼"이라고 쳐도 나오고, "클릭"이라고 쳐도 Button이 나오도록 했다. 검색은 사용자가 정확한 이름을 몰라도 찾을 수 있어야 한다.

키보드 단축키 ⌘K는 macOS 사용자라면 반사적으로 누른다. 이 단축키 하나로 사이트가 "프로페셔널한 도구"처럼 느껴진다. 사소한 것 같지만, 사용자가 사이트를 처음 열었을 때 받는 첫인상을 완전히 바꾼다.

---

## 다크 모드: CSS 변수 60개의 의미

다크 모드를 제대로 지원하려면 색상을 하드코딩하면 안 된다. 처음부터 CSS custom properties로 설계해야 한다.

나는 색상 토큰을 시맨틱 레이어로 나눴다.

- **Primitive tokens**: `--color-gray-100`, `--color-blue-500` 같은 원시 값
- **Semantic tokens**: `--color-bg-primary`, `--color-text-secondary`, `--color-border-subtle` 같은 의미 기반 값
- **Component tokens**: `--button-bg-hover`, `--card-shadow` 같은 컴포넌트 전용 값

다크 모드 전환은 `html` 엘리먼트에 `data-theme="dark"` 속성을 붙이고, CSS에서 `[data-theme="dark"]` 셀렉터로 시맨틱 토큰 값만 재정의하는 방식이다. Primitive tokens는 건드리지 않는다.

결과적으로 60개 이상의 CSS 변수가 다크 모드에서 재정의된다. 처음에는 많다고 느껴졌지만, 이게 맞는 방법이다. 변수가 충분히 세분화되어야 컴포넌트 어디서든 정확한 색상 토큰을 쓸 수 있다.

사용자 설정은 `localStorage`에 저장한다. 다음에 들어올 때 마지막 테마가 유지된다. 이 작은 디테일이 없으면 사용자는 매번 다크 모드를 켜야 한다. 기억해주지 않는 사이트는 불친절하다.

## Blocks 페이지: 미니어처 렌더링

shadcn/ui의 Blocks 페이지에서 영감을 받았다. 컴포넌트 조합으로 만든 실제 UI 패턴들을 미리보기 형태로 보여주는 페이지다.

핵심 아이디어는 실제 컴포넌트를 작은 iframe 안에 렌더링하는 것이다. 스크린샷이 아니라 살아있는 UI. 사용자는 미리보기를 클릭하면 전체 화면으로 볼 수 있고, "코드 복사" 버튼으로 바로 가져갈 수 있다.

Blocks 페이지가 중요한 이유는 컴포넌트 문서와 다른 레이어를 보여주기 때문이다. 컴포넌트 문서는 "이 버튼의 Props는 뭔가요?"라는 질문에 답한다. Blocks 페이지는 "실제 화면을 어떻게 조합하나요?"라는 질문에 답한다. 이 두 레이어가 모두 있어야 문서가 완성된다.

---

## 인사이트

문서 사이트를 하루에 만들 수 있었던 이유는 기술력이 아니었다. 좋은 레퍼런스를 골랐고, 그 레퍼런스의 핵심 패턴만 뽑아서 우리 것으로 만들었기 때문이다.

레퍼런스 선택이 80%다. 나머지 20%는 실행이다.

---

## 패턴

- **레이아웃은 빌리고, 정체성은 직접 만든다**: Carbon의 구조 + 우리의 토큰
- **검색은 첫날부터**: 나중에 붙이려면 구조를 다 바꿔야 한다
- **CSS 변수는 충분히 많아야 한다**: 60개가 많아 보여도, 각자 역할이 있다
- **두 레이어의 문서**: 컴포넌트 레퍼런스 + 실제 사용 패턴(Blocks)
