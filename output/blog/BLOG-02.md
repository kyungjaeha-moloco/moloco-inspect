# 내가 설계하고 AI가 구현한다 — Claude Code와의 페어 작업 실전기

## TL;DR

- 나는 아키텍처와 UX를 결정했다. Claude는 코드를 썼다. 이 분업이 18일을 가능하게 했다.
- Session Handoff 패턴: AI 세션이 끊겨도 맥락이 이어지도록 문서를 설계했다.
- AI가 쓴 코드를 다른 AI가 리뷰하는 Codex Review를 운영했다.

---

## "같이 일한다"는 게 무슨 의미인가

처음에 나는 Claude Code를 "코드 생성 도구"로 생각했다. 내가 뭘 원하는지 말하면 코드를 뽑아주는 것. 그런데 18일을 같이 일하고 나서 생각이 바뀌었다. 이건 페어 프로그래밍에 더 가깝다. 단, 내가 페어 중 한 명인데 코딩을 못 한다는 게 다를 뿐이다.

페어 프로그래밍에서 한 명이 운전하고 한 명이 내비게이션을 한다고 한다. 나는 내비게이션이었다. 어디로 가야 하는지, 왜 이 방향이 맞는지, 다음 교차로에서 어떻게 꺾어야 하는지. Claude는 실제로 운전대를 잡았다.

이 분업이 잘 작동하려면 내비게이션이 정확해야 한다. 그게 내 역할의 본질이었다.

## 내가 한 것 vs AI가 한 것

명확하게 나눠보면 이렇다.

**내가 한 것:**
아키텍처 결정. Chrome Extension이 캡처하는 데이터의 스키마를 정의하는 건 내가 했다. 어떤 DOM 정보를 추출할지, 그 정보를 Orchestrator에 어떻게 넘길지. 이건 제품의 "무엇"에 관한 결정이었고, 내가 UX 관점에서 판단했다.

UX 흐름 설계. PM이 크롬 익스텐션을 열어서 UI 요소를 선택하고, 자연어로 요청을 입력하고, 프리뷰를 보고, PR을 올리는 전체 흐름. 각 단계에서 사용자가 무엇을 보고 느껴야 하는지. 이건 Figma 없이 텍스트로 설계했다.

프롬프트 작성. AI 에이전트에게 코드 수정을 지시하는 프롬프트는 내가 설계했다. "디자인 시스템 컨텍스트를 먼저 읽고, 변경 범위를 최소화하고, 관련 없는 코드는 건드리지 마라"는 제약 조건들. 이게 제품 품질을 결정했다.

**AI가 한 것:**
실제 코드 구현. Node.js Orchestrator의 API 엔드포인트, Docker 컨테이너 설정, Chrome Extension의 content script, Dashboard의 상태 관리. 이 모든 코드는 Claude가 썼다.

디버깅. 뭔가 동작하지 않으면 에러 메시지를 가져다 주고 "왜 이런 일이 생기는지, 어떻게 고칠지"를 물었다. 대부분 Claude가 원인을 진단하고 수정안을 제시했다.

리팩토링. 동작하는 코드가 생기면 "이걸 더 깔끔하게 만들어줘"라고 했다. 중복 제거, 함수 분리, 에러 처리 추가.

## Session Handoff: 가장 중요한 패턴

Claude Code는 세션이 끊기면 이전 대화를 기억하지 못한다. 처음엔 이게 큰 문제로 보였다. 아침에 설계한 것을 오후 세션에서 이어가려면 다시 설명해야 하는가?

그래서 Session Handoff 패턴을 만들었다.

매일 작업이 끝나면 `.sisyphus/plans/` 폴더에 핸드오프 문서를 썼다. 형식은 이렇다.

- **오늘 완료한 것**: 어떤 결정을 내렸고, 어떤 코드가 생겼고, 어떤 이유로 그렇게 했는지
- **현재 상태**: 지금 코드베이스의 상태. 어떤 파일이 어떤 역할을 하는지
- **미해결 문제**: 내일 시작할 때 알아야 할 이슈들
- **다음 세션의 첫 번째 작업**: 구체적으로 무엇을 해야 하는지

이 문서를 다음 세션 시작 때 Claude에게 읽게 했다. "이 문서를 읽고 현재 상태를 파악한 다음, 다음 작업부터 시작해줘." 30초 만에 맥락이 전달됐다.

처음에는 이걸 쓰는 게 번거롭다고 생각했다. 그런데 쓰다 보니 이 문서가 나 자신을 위한 것이기도 했다. 다음 날 아침 내가 어제 어디까지 했는지를 정확히 알 수 있었다.

## Codex Review: AI가 AI를 리뷰한다

코드 리뷰를 누가 하는가라는 문제가 생겼다. 나는 코드를 잘 모른다. 그렇다고 Claude가 자기가 쓴 코드를 스스로 리뷰하는 건 한계가 있다.

그래서 Codex Review 패턴을 썼다. Claude가 코드를 쓰고 나면, 새 세션을 열어서 그 코드를 가져다주고 리뷰를 요청했다. 새 세션의 Claude는 이전 세션의 맥락이 없기 때문에 코드를 순수하게 본다.

리뷰 요청은 구체적으로 했다. "보안 취약점이 있는가", "에러 처리가 빠진 곳이 있는가", "이 코드가 내가 의도한 대로 동작하는가"를 각각 물었다. 추상적인 "잘 됐어?"는 답도 추상적이었다.

이 방식으로 실제 문제 몇 가지를 잡았다. Docker 샌드박스에서 파일 권한 처리가 누락된 것, Orchestrator API에서 타임아웃 처리가 불완전한 것. 내가 코드를 읽었다면 발견하지 못했을 것들이다.

## 프롬프트 작성이 핵심 스킬이다

18일 동안 가장 많이 발전한 스킬은 프롬프트 작성이었다. 초반에는 "X를 만들어줘"처럼 요청했다. 결과가 종종 내가 원하는 것과 달랐다.

시간이 지나면서 프롬프트 구조가 생겼다.

```
[맥락] 이 시스템은 Chrome Extension이 캡처한 DOM 정보를 Orchestrator가 받아서...
[제약] Docker 샌드박스 안에서만 코드를 수정해야 하고, 디자인 시스템 JSON을 참고해야 해
[요청] 지금 필요한 건 Orchestrator의 /analyze 엔드포인트야
[기준] 성공 조건은 Chrome Extension에서 POST 요청을 보냈을 때 sandbox task ID가 반환되는 것
```

맥락, 제약, 요청, 성공 기준. 이 네 가지가 들어가면 결과가 훨씬 정확해졌다.

이건 사실 UX 설계와 다르지 않다. 사용자가 누구인지, 어떤 제약 안에서 작업하는지, 무엇을 원하는지, 성공이 무엇인지. 프롬프트 작성은 요구사항 정의의 다른 형태다.

## 동작하지 않을 때 내가 한 일

AI와 일하다 보면 막히는 순간이 온다. 코드를 줬는데 동작을 안 한다. 수정을 요청했는데 더 이상해졌다.

이때 내가 배운 것: 문제를 더 작게 자른다.

"이 전체 시스템이 왜 안 돼?"라고 물으면 Claude도 제대로 된 답을 못 한다. "Docker 컨테이너가 시작됐는지 확인하는 명령어가 뭐야"처럼 한 단계씩 쪼개서 물으면 원인을 찾을 수 있었다.

이것도 결국 설계 능력이다. 복잡한 문제를 작은 단위로 분해하는 것. 엔지니어들이 디버깅할 때 하는 것과 같은 사고 과정이다. 나는 코드를 못 쓰지만 이 사고 과정은 할 수 있었다.

## 재현 가능한 패턴

18일 동안 굳어진 협업 패턴을 정리하면 이렇다.

첫째, 나는 "무엇"을, AI는 "어떻게"를. 결정권의 경계를 명확하게.

둘째, Session Handoff를 매일 쓴다. AI의 메모리 한계를 문서로 보완한다.

셋째, 새 세션에서 코드 리뷰를 받는다. 동일 세션의 자기 리뷰는 한계가 있다.

넷째, 프롬프트에 맥락/제약/요청/기준을 모두 넣는다.

다섯째, 막히면 더 작게 자른다. 범위를 좁힐수록 AI의 답이 정확해진다.

이 다섯 가지가 내 AI 페어 작업의 실전 규칙이 됐다.
