# 'Working'이냐 'Coding'이냐 — PM이 보는 UI의 단어 선택

## TL;DR

- Stepper 레이블을 "Coding"에서 "Working"으로 바꿨다. 사용자에게 AI가 무엇을 하는지 설명할 때, 기술 용어보다 행동 용어가 더 정직하다.
- "Approve & Apply" 버튼을 "✓ Approve & Create PR"로 바꿨다. 같은 행동이지만 기대값이 달라진다.
- 마이크로카피 하나가 제품의 신뢰도를 만든다. 코드보다 카피를 더 오래 고민한 날이 있었다.

---

## 배경: 4단계 스테퍼를 만들면서 생긴 고민

Chrome Extension 사이드패널에는 진행 상태를 보여주는 4단계 스테퍼가 있다. 사용자가 UI 요소를 선택하고 변경 요청을 보내면, 이 스테퍼가 현재 어느 단계에 있는지 실시간으로 보여준다.

처음에 만든 레이블은 이렇게 생겼다.

```
Setup → Coding → Verify → Done
```

PM 입장에서 보면 자연스러운 네이밍이다. AI가 코드를 수정하고 있으니 "Coding"이 맞는 것 같다. 엔지니어에게 물어봐도 아마 "맞는 거 아닌가요?"라고 할 것이다.

그런데 나는 며칠 동안 이 레이블이 마음에 걸렸다. 제품 디자이너로서 단어 하나에 이렇게 오래 걸릴 줄은 몰랐다.

---

## "Coding"의 문제

이 제품의 사용자는 PM이다. 코드 리뷰를 하지 않는 사람, 터미널을 열지 않는 사람, IDE가 뭔지는 알지만 매일 쓰지는 않는 사람.

"Coding" 단계가 진행 중이라는 표시를 보면서 PM은 무슨 생각을 할까?

아마 "AI가 코드를 짜고 있구나"라고 생각할 것이다. 그게 맞는 생각이다. 하지만 그게 PM에게 필요한 정보인가?

아니다. PM이 알아야 하는 건 "지금 무언가가 처리되고 있다"는 것이다. 그 '무언가'가 코드 수정인지, 분석인지, 파일 탐색인지는 중요하지 않다. 오히려 "Coding"이라는 단어는 PM에게 "이건 개발자 영역의 일"이라는 인상을 주고, 자신이 개입해서는 안 되는 블랙박스처럼 느껴지게 만든다.

더 솔직하게 말하면: AI 에이전트가 "코딩"만 하는 게 아니다. 파일을 탐색하고, 컴포넌트 구조를 분석하고, 디자인 시스템 규칙을 확인하고, 그 결과를 바탕으로 코드를 작성한다. 이 복잡한 과정을 "Coding"이라고 부르는 건 과소 표현이면서 동시에 오해를 부른다.

그래서 바꿨다.

```
Setup → Working → Verify → Done
```

"Working"은 더 포괄적이다. 그리고 더 정직하다. AI가 지금 무언가를 열심히 하고 있다는 것. 그게 이 단계에서 사용자에게 전달해야 하는 전부다.

코드를 한 줄도 바꾸지 않았다. 레이블 텍스트 하나를 바꿨을 뿐인데, 제품이 PM에게 말하는 방식이 달라졌다.

---

## "Approve & Apply" 버튼의 실패

버튼 텍스트도 비슷한 고민을 했다.

처음 만든 버튼은 "Approve & Apply"였다. AI가 수정한 코드를 확인하고 승인하면, 변경사항이 적용된다. 논리적으로는 맞다.

그런데 실제로 버튼을 눌렀을 때 일어나는 일은 이렇다: 브랜치가 생성되고, 커밋이 올라가고, GitHub에 PR이 만들어진다. PM은 PR 링크를 받고, 코드 리뷰를 요청하는 메시지를 팀에 보낼 수 있다.

"Apply"라는 단어에서 이 흐름이 보이는가? 전혀 보이지 않는다. "Apply"는 "지금 바로 코드베이스에 적용된다"는 인상을 준다. 두려운 버튼이다. 누르기 전에 세 번쯤 망설이게 된다.

반면 사용자가 실제로 경험하는 것은 그렇지 않다. 변경사항은 즉시 메인 브랜치에 들어가지 않는다. PR을 통해 리뷰 과정을 거친다. 이건 PM이 이미 아는 흐름이다.

그래서 바꿨다.

```
✓ Approve & Create PR
```

체크마크가 먼저 나온다. 승인한다는 행위의 성격을 시각적으로 표시한다. 그리고 "Create PR"은 결과를 정확히 설명한다. 코드가 바로 배포되는 게 아니라, PR이 만들어진다. 팀의 리뷰를 거치는 프로세스가 남아 있다.

같은 기능이다. 하지만 버튼을 누르는 사람의 심리가 달라진다. "Approve & Apply"는 불안감을 준다. "✓ Approve & Create PR"은 익숙한 흐름의 시작점을 제안한다.

---

## Inspect Agent — AI에게 이름을 붙이다

또 다른 마이크로카피 결정이 있었다. AI가 생성한 실행 계획을 어떻게 보여줄 것인가.

Claude Sonnet이 사용자의 요청을 분석하고, 어떤 파일을 수정할지, 어떤 위험이 있는지, 어떻게 검증할지를 생성한다. 이 분석 결과를 UI에 표시해야 한다.

처음에는 그냥 "AI Analysis"라는 헤더 아래 결과를 나열했다. 기능적으로는 동작한다. 하지만 뭔가 냉랭하다. "AI가 분석했습니다"라는 말은 보고서 느낌이다. 사용자와 AI 사이의 거리가 느껴진다.

그래서 Inspect Agent라는 이름을 붙였다.

```html
<span class="plan-ai-name">Inspect Agent</span>
```

사이드패널에서 Inspect Agent는 사용자에게 말을 건다. "이 요청을 이렇게 이해했습니다. 이런 접근으로 수정하겠습니다. 이런 위험이 있을 수 있습니다." 보고서가 아니라 대화다.

AI 출력에 인격을 부여하면 신뢰도가 올라간다. 이건 내 직관이 아니라 경험에서 나온 관찰이다. 같은 내용이어도 "AI 분석 결과"보다 "Inspect Agent의 계획"이 더 읽힌다. 더 책임감 있어 보인다. 잘못됐을 때도 "AI가 틀렸다"보다 "Inspect Agent가 이 부분을 놓쳤다"가 더 구체적인 피드백을 유도한다.

이름을 붙이는 건 단순한 마케팅이 아니다. AI를 도구가 아닌 팀원으로 포지셔닝하는 UX 전략이다.

---

## 핵심 인사이트

마이크로카피는 기능이 아니다. 기대값을 설정하는 인터페이스다.

사용자는 버튼을 누르기 전에 그 버튼이 무슨 일을 할지 예측한다. 그 예측이 현실과 맞으면 신뢰가 생기고, 어긋나면 불안이 생긴다. "Coding"과 "Working"의 차이는 글자 수가 아니다. 사용자에게 무엇을 기대하게 만드느냐의 차이다.

PM에게는 "AI가 코딩 중"이라는 정보보다 "AI가 작업 중"이라는 정보가 더 안심이 된다. 기술적 정확성보다 심리적 정확성이 더 중요한 순간이 있다.

---

## 재현 가능한 패턴

버튼이나 레이블을 만들 때 이 세 가지를 확인한다.

첫째, 이 텍스트를 읽는 사람이 누구인가. 엔지니어에게 맞는 언어와 PM에게 맞는 언어는 다르다.

둘째, 이 텍스트가 설정하는 기대값은 무엇인가. 사용자가 이 텍스트를 읽고 무슨 일이 일어날 것이라고 예상하는가. 그 예상이 현실과 얼마나 일치하는가.

셋째, 결과를 설명하는가 아니면 과정을 설명하는가. 대부분의 경우 사용자에게는 결과가 더 중요하다. "Create PR"이 "Apply"보다 낫고, "Working"이 "Coding"보다 낫다.

단어 하나가 제품의 신뢰를 만든다.
